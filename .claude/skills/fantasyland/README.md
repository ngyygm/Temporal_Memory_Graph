# Fantasyland — 基于 Deep-Dream 的长程叙事创作引擎

> **核心理念：写前大量检索记忆 → 写初稿 → 逐句多轮校验 → 全部通过才输出定稿并存入。**
>
> AI 长篇创作最大的敌人不是文笔，而是**吃书**——写到第五章忘了第三章的设定，角色性格前后矛盾，世界观越写越崩。Fantasyland 把 Deep-Dream 的时序记忆图谱作为叙事的"外部大脑"，写前大量检索所有相关记忆，写完后逐句多轮迭代校验，只有全部通过才输出定稿并存入记忆，让整部作品始终锚定在自己的设定上。
>
> 适用于：小说、故事、连载、真实事件推演等任何需要长程一致性的叙事创作。

## 问题：为什么 AI 写小说会吃书

| 根因 | 表现 | Fantasyland 的解法 |
|------|------|-------------------|
| 上下文窗口有限 | 写到后期，前文设定被截断 | 每次写作前从记忆图谱**按需检索**，不需要塞入全部前文 |
| 信息过载 | 塞入大量前文后 LLM 注意力涣散 | 只检索**当前段落相关的**实体和关系，上下文精简 |
| 无状态累积 | 每次会话从零开始，不知道之前写了什么 | 所有内容写入记忆图谱，**永久可查** |
| 时间线混乱 | 角色第 3 章受的伤到第 5 章突然好了 | 时序图谱记录**版本变更链**，可追溯任意时间点状态 |
| 设定随意扩展 | 写着写着加了新设定，和原有设定冲突 | 新内容写入前经过**一致性校验** |


Fantasyland 本身是一个 **Agent Loop**，不是一次性生成整章，而是：

1. 每写一段（50-300 字），先调用 Deep-Dream 检索相关记忆
2. 基于检索到的上下文写作
3. 写完后校验这段内容是否和已有记忆矛盾
4. 确认无误后存入记忆图谱
5. 进入下一段

## 创作循环（The Loop）

### Phase 0: 启动准备

**目标：** 初始化记忆图谱，创建专属的小说知识空间。

```
graph_id = "novel_{book_name}"     # 每部小说独立图谱
```

**Deep-Dream 调用：**
- `graph_summary` → 确认图谱状态（空图则开始构建，非空则继续创作）

---

### Phase 1: 世界观构建

**目标：** 把小说的基础设定全部写入记忆图谱，作为后续一切创作的"锚点"。

#### 1.1 输入设定

由创作者提供（或 AI 辅助生成后经确认）以下设定内容：

| 设定类型 | 示例 |
|---------|------|
| **角色** | 姓名、外貌、性格、背景故事、能力、弱点 |
| **世界观** | 时代背景、地理、社会结构、种族 |
| **力量体系** | 魔法/武功/科技体系的规则和限制 |
| **核心冲突** | 主线矛盾、阵营对立 |
| **重要物品** | 关键道具及其属性 |
| **组织势力** | 公会/门派/国家/家族 |
| **历史事件** | 故事开始前的关键历史 |

#### 1.2 结构化写入记忆

对每类设定，**不要一次性丢给 remember**，而是按维度分批写入，确保实体抽取质量：

```
# 每个角色单独写入
remember(text="角色：林嘿嘿。考古学博士，28岁，性格沉稳但偶尔冲动。
         外貌：瘦高，戴圆框眼镜，左手有一道小时候留下的疤痕。
         背景：父母早逝，由师父抚养长大，师门为青山派。
         弱点：怕蛇，对师父的死一直心怀愧疚。",
         source_name="设定-角色-林嘿嘿",
         wait=true)

# 世界观设定
remember(text="世界观：大陆分为东域和西域，中间由万丈深渊隔开。
         东域以武道为主，西域以魔法为主。
         当前时间线：东历1247年，距上一次深渊裂隙事件已过300年。",
         source_name="设定-世界观",
         wait=true)

# 力量体系
remember(text="力量体系：武道分为九品，一品最低，九品最高。
         突破瓶颈需要领悟'道意'。
         禁忌：强行突破会走火入魔，轻则武功全废，重则当场死亡。",
         source_name="设定-力量体系",
         wait=true)
```

#### 1.3 验证设定完整性

写入后立刻验证：

```
# 检查关键角色是否被正确抽取
quick_search(query="林嘿嘿的身份和背景")

# 检查世界观是否完整
ask(question="这个世界有哪些域？力量体系是怎样的？")

# 检查角色间关系是否被识别
get_relations_between(entity_a="林嘿嘿", entity_b="青山派")
```

如果发现缺失，用 `create_entity` / `create_relation` 手动补全。

### Phase 2: 大纲规划

**目标：** 制定全书的叙事结构，写入记忆作为写作的路标。

#### 2.1 生成大纲前：读取已有设定

```
# 先回忆所有设定
graph_summary          → 了解图谱规模
quick_search(query="主要角色和他们的关系")
quick_search(query="世界观和力量体系的核心规则")
```

#### 2.2 大纲层次

```
全书大纲
├── 总纲 (整体故事弧: 起承转合)
│   → 记为 source_name="大纲-总纲"
├── 卷纲 (每卷的核心冲突和走向)
│   → 记为 source_name="大纲-第X卷"
├── 章纲 (每章的目标、场景、出场角色)
    → 记为 source_name="大纲-第X章"
```

#### 2.3 写入大纲

```
remember(text="""
第3章「深渊之下」大纲：
- 目标：林嘿嘿和白狐首次深入万丈深渊
- 场景：深渊边缘 → 悬崖栈道 → 地下暗河
- 出场角色：林嘿嘿、白狐、（暗线）神秘追踪者
- 核心冲突：白狐对深渊有恐惧，林嘿嘿需要取得它的信任
- 章末钩子：在暗河中发现一块刻有青山派标志的石碑
- 伏笔：石碑上的日期比青山派建派时间早200年
""",
         source_name="大纲-第3章",
         wait=true)
```

#### 2.4 用 Dream 发现隐藏关联

大纲写完后，用 `dream_run` 让系统自主探索设定间的隐藏联系，为后续伏笔设计提供灵感：

```
dream_run(strategy="cross_domain", seed_count=5, max_depth=3)
dream_run(strategy="narrative", seed_count=3, max_depth=2)
```

Dream 发现的关系（如 "青山派石碑与万丈深渊有关联"）会自动记录，后续写作时可作为伏笔素材。

### Phase 3: 章节循环

**这是 Fantasyland 的核心循环。** 每一章的写作流程如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                     章节循环 (Chapter Loop)                      │
│                                                                  │
│  Step 1: 章节预设                                                │
│  ┌──────────────┐                                               │
│  │ quick_search │ 读取本章大纲 + 前文最新状态                     │
│  └──────┬───────┘                                               │
│         ▼                                                        │
│  Step 2: 段落循环 (Paragraph Loop)                               │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │  2a. 记忆召回                                            │  │
│  │  ┌────────────┐                                         │  │
│  │  │quick_search│ 根据"接下来要写的场景"检索相关记忆          │  │
│  │  │ ask        │ 提问确认角色当前状态、已有伏笔等           │  │
│  │  └─────┬──────┘                                         │  │
│  │        ▼                                                 │  │
│  │  2b. 写一段 (50-300字)                                    │  │
│  │  ┌────────────┐                                         │  │
│  │  │ LLM 创作    │ 基于检索到的记忆上下文，写一段正文         │  │
│  │  └─────┬──────┘                                         │  │
│  │        ▼                                                 │  │
│  │  2c. 一致性校验                                           │  │
│  │  ┌────────────┐    ┌────────────┐                        │  │
│  │  │ remember   │───▶│ 校验规则    │                        │  │
│  │  │ 存入临时    │    │ (见下方)    │                        │  │
│  │  └────────────┘    └─────┬──────┘                        │  │
│  │                          │                                │  │
│  │                  ┌───────┴────────┐                      │  │
│  │                  │                │                      │  │
│  │               通过 ✓          矛盾 ✗                     │  │
│  │                  │                │                      │  │
│  │                  ▼                ▼                      │  │
│  │            确认写入          修改这一段                    │  │
│  │            进入下一段        重新校验                      │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  Step 3: 章节收尾                                                │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐        │
│  │ remember     │   │ dream_run    │   │ 更新版本     │        │
│  │ 存储完整章节  │   │ 发现新关联   │   │ 角色状态变更  │        │
│  └──────────────┘   └──────────────┘   └──────────────┘        │
│                                                                  │
│  → 进入下一章                                                    │
└─────────────────────────────────────────────────────────────────┘
```

#### Step 1: 章节预设

每章开始前，检索本章大纲和前文的最新进展：

```python
# 读取本章大纲
quick_search(query="第3章大纲和核心冲突")

# 读取上一章结尾，了解角色当前位置和状态
ask(question="上一章结束时，林嘿嘿在哪里？他的状态如何？白狐和他在一起吗？")

# 读取所有未回收的伏笔
quick_search(query="已埋下的伏笔和未解之谜")

# 读取本章涉及角色的最新状态
batch_profiles(family_ids=["ent_linheihei", "ent_baihu"])
```

基于检索结果，生成**本章场景预设**（不是正文，是写作指南）：

```
remember(text="""
第3章场景预设：
- 开场：深渊边缘，清晨雾气中，林嘿嘿检查装备
- 角色状态：林嘿嘿（六品武者，左臂有轻伤），白狐（道行500年，对深渊有阴影）
- 情绪基调：紧张中带一丝冒险的兴奋
- 本章必须推进的伏笔：石碑上的日期之谜
- 本章需要埋下的新伏笔：暗河深处的低沉回声
""",
         source_name="预设-第3章",
         wait=true)
```

#### Step 2: 段落循环

**这是核心中的核心——逐段写作、逐段校验。**

##### 2a. 记忆召回（每段写作前）

根据即将写作的内容，精确检索需要的记忆：

```
# 场景类检索：当前场景的相关信息
quick_search(query="万丈深渊的地理环境和已知信息")

# 角色类检索：出场角色的当前状态
ask(question="白狐为什么害怕深渊？它在三百年前经历过什么？")

# 物品类检索：角色当前拥有的道具
quick_search(query="林嘿嘿目前携带的物品和装备")

# 关系类检索：角色间当前的关系状态
get_relations_between(entity_a="林嘿嘿", entity_b="白狐")
```

##### 2b. 写一段正文

基于检索到的记忆上下文，**只写一小段**（50-300字），而不是一整章。

原则：
- 每段只推进一个小事件或一个场景细节
- 涉及角色动作/对话时，必须基于记忆中的性格和行为模式
- 描述环境时，必须和记忆中的世界观设定一致

##### 2c. 一致性校验

写完一段后，**不要急着写下一段**，先校验：

```
# 方法1：用 ask 让系统自动检查
ask(question="以下内容是否和已有设定矛盾：'林嘿嘿用右手拔出了师父留下的青钢剑'？
    请检查林嘿嘿是否惯用右手、他是否拥有青钢剑、青钢剑是否来自师父。")

# 方法2：检索相关实体，人工比对
quick_search(query="林嘿嘿的惯用手和武器")
quick_search(query="青钢剑的来历和归属")

# 方法3：用矛盾检测
entity_profile(family_id="ent_linheihei")
# 查看 contradictions
```

**校验维度清单：**

| 维度 | 检查内容 | 示例 |
|------|---------|------|
| 角色属性 | 外貌、性格、能力是否和设定一致 | 设定说左手有疤，文中不能写"双手完好" |
| 角色状态 | 当前受伤/装备/情绪是否和前文衔接 | 上章刚受了重伤，这章不能活蹦乱跳 |
| 世界观 | 地理、规则、力量体系是否合规 | 设定说突破需要道意，不能凭空突破 |
| 时间线 | 事件顺序是否合理 | 白天发生的事不能在日落前被当作"昨天" |
| 因果链 | 行为的动机是否合理 | 角色不能突然知道他不可能知道的信息 |
| 伏笔状态 | 未回收伏笔不能遗忘 | 第1章埋的线索到第5章必须有呼应或延续 |

#### Step 3: 章节收尾

一章完成后：

```
# 1. 存储完整章节内容
remember(text=chapter_full_text,
         source_name="正文-第3章-深渊之下",
         wait=true)

# 2. 显式记录本章角色状态变更
remember(text="""
第3章结束状态变更：
- 林嘿嘿：发现石碑后陷入沉思，对师门历史产生怀疑
- 白狐：克服了对深渊的部分恐惧，与林嘿嘿的信任加深
- 新发现：石碑上刻有'青山'二字，日期为东历1047年
- 未解之谜：暗河深处传来的回声来源不明
""",
         source_name="状态-第3章结束",
         wait=true)

# 3. 用 Dream 发现本章内容与已有设定的新关联
dream_run(strategy="narrative", seed_count=3)

# 4. 更新角色实体的摘要（反映最新状态）
evolve_entity_summary(family_id="ent_linheihei")
evolve_entity_summary(family_id="ent_baihu")
```

### Phase 4: 伏笔与趣味性设计

**目标：** 让小说不只是"不吃书"，还要**好看**。

#### 4.1 伏笔系统

伏笔在记忆图谱中以**关系**的形式存在，有三个阶段：

```
  埋设 (Planted)          铺垫 (Developed)         回收 (Resolved)
  第1章埋下线索    →    第3章加深暗示     →    第7章揭晓真相
       │                       │                       │
  relation.status:        relation.status:        relation.status:
  "planted"               "developed"             "resolved"
```

**埋设伏笔：**

```
# 写作时发现可以埋的伏笔 → 立刻记录
remember(text="""
伏笔-001：暗河回声
- 类型：悬念
- 埋设章节：第3章
- 内容：林嘿嘿在暗河深处听到规律的回声，像是某种信号
- 预计回收：第7-8章
- 回声实际上是远古封印松动的前兆
- 相关实体：万丈深渊、暗河、远古封印
""",
         source_name="伏笔-001",
         wait=true)
```

**追踪伏笔状态：**

```
# 写每章前检查未回收伏笔
quick_search(query="所有未回收的伏笔")

# 写到预计回收章节时
ask(question="有哪些伏笔预计在这一章回收？各自的铺垫情况如何？")
```

**用 Dream 发现伏笔机会：**

```
# 跨域发现：哪些看似无关的元素可以建立联系
dream_run(strategy="cross_domain", seed_count=5, max_depth=3)

# 对比发现：相似元素的差异可以制造反转
dream_run(strategy="contrastive", seed_count=4)

# 叙事发现：碎片化情节的串联可能
dream_run(strategy="narrative", seed_count=3)
```

#### 4.2 趣味性增强策略

| 策略 | 实现方式 | 记忆查询 |
|------|---------|---------|
| **悬念层叠** | 每解决一个谜题，引出更大的谜题 | `quick_search(query="当前未解之谜")` |
| **角色弧光** | 让角色在经历中成长/改变，并记录变化轨迹 | `get_entity_timeline(family_id)` |
| **多线叙事** | 用 Dream 发现可以交叉的情节线 | `dream_run(strategy="cross_domain")` |
| **反转设计** | 对比设定中的矛盾点，设计合理反转 | `get_entity_contradictions(family_id)` |
| **情感锚点** | 在关键情节前后强化情感描写 | `ask(question="林嘿嘿对师父的感情线索")` |
| **信息差** | 读者知道但角色不知道（或反过来）的信息 | `traverse_graph(seed, max_depth=2)` |

#### 4.3 节奏控制

每章写作前检查叙事节奏：

```
ask(question="前几章的叙事节奏如何？是否需要调整？
    近几章的冲突密度、情感密度、动作密度分别是多少？")
```

建议节奏模式：

```
紧张章 → 渗透章（伏笔铺垫）→ 爆发章 → 回响章（消化+新伏笔）
   ↑                                           │
   └───────────────────────────────────────────┘
```

## 记忆交互协议

Fantasyland 在写作过程中对 Deep-Dream 的调用遵循固定协议：

### 写作前（必须）

```
1. quick_search  → 召回当前场景相关的实体和关系
2. ask           → 确认角色当前状态和位置
3. quick_search  → 检查相关伏笔的当前状态
```

### 写作中（每段）

```
4. [LLM 创作一段文字]
```

### 写作后（必须）

```
5. ask / quick_search → 一致性校验
6. remember          → 存储写好的段落（source_name 标注章节和段号）
```

### 章节完成时（必须）

```
7. remember          → 存储完整章节
8. remember          → 记录状态变更摘要
9. evolve_entity_summary → 更新关键角色的最新状态
10. dream_run        → 发现新关联，为后续章节提供素材
```

### 调用频率参考

| 阶段 | 每章调用次数 | 主要工具 |
|------|------------|---------|
| 章节预设 | 3-5 | quick_search, ask, batch_profiles |
| 段落写作 | 每段 3-6 | quick_search, ask, remember |
| 章节收尾 | 3-5 | remember, dream_run, evolve_entity_summary |
| **每章总计** | **约 20-40** | |

## 不吃书校验机制

### 三层校验

```
┌─────────────────────────────────────────────────┐
│           Layer 1: 实体属性校验 (自动)            │
│                                                   │
│  写到角色/物品/地点时 → quick_search 查询原文设定   │
│  比对当前描述是否和实体记忆一致                      │
│  不一致 → 修改这一段                               │
└───────────────────────┬─────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────┐
│           Layer 2: 状态连续性校验 (自动)           │
│                                                   │
│  写到角色行为时 → ask 确认角色当前状态               │
│  检查行为是否和当前身体/情绪/装备状态匹配            │
│  如：受伤的手臂不能灵活挥剑                         │
└───────────────────────┬─────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────┐
│           Layer 3: 叙事逻辑校验 (半自动)           │
│                                                   │
│  写到关键情节转折时 → 检查因果链是否合理             │
│  检查信息获取是否合理（角色不能知道他不知道的事）      │
│  检查时间线是否连续                                 │
│  这一层的部分判断需要创作者介入确认                   │
└─────────────────────────────────────────────────┘
```

### 自动校验 Prompt 模板

每段写完后，系统自动执行以下校验：

```
ask(question="""
请检查以下段落是否和记忆图谱中的已有信息矛盾：

【待校验段落】
{paragraph_text}

请逐一检查：
1. 角色属性（外貌、性格、能力）是否一致
2. 角色当前状态（受伤、装备、位置）是否和前文衔接
3. 世界观设定（地理、规则）是否合规
4. 时间线是否连续合理
5. 因果关系是否成立

如果发现矛盾，请指出具体矛盾点和记忆中的原文。
""")
```

## 配置与使用

### 前置条件

1. Deep-Dream 服务已启动 (`http://localhost:16200`)
2. Neo4j 或 SQLite 存储后端已配置
3. LLM 和 Embedding 服务已配置

### 创建小说图谱

```bash
# 为每部小说创建独立图谱（通过 graph_id 区分）
# 所有 API 调用带上 graph_id 参数即可

# 例：
curl -s -X POST $BASE_URL/remember \
  -H 'Content-Type: application/json' \
  -d '{"graph_id":"novel_my_book","text":"...","wait":true}'
```

### 写作流程命令速查

```
# ===== Phase 1: 世界观构建 =====
remember(world_settings, source_name="设定-世界观", wait=true)
remember(character_details, source_name="设定-角色-XXX", wait=true)
remember(power_system, source_name="设定-力量体系", wait=true)
quick_search(query="验证设定是否完整")

# ===== Phase 2: 大纲 =====
graph_summary                           # 了解已有设定规模
quick_search(query="主要角色和关系")
remember(story_outline, source_name="大纲-总纲", wait=true)
remember(chapter_outline, source_name="大纲-第N章", wait=true)
dream_run(strategy="cross_domain")      # 发现隐藏关联

# ===== Phase 3: 章节循环 =====
# --- 章节预设 ---
quick_search(query="第N章大纲")
ask(question="上一章结束时各角色的状态")
quick_search(query="未回收的伏笔")
remember(chapter_preset, source_name="预设-第N章", wait=true)

# --- 段落循环 (每段重复) ---
quick_search(query="当前场景相关信息")    # 写前检索
ask(question="角色当前状态确认")
# [LLM 写一段]
ask(question="校验这段内容是否矛盾")       # 写后校验
remember(paragraph, source_name="正文-第N章-P{n}", wait=true)

# --- 章节收尾 ---
remember(chapter_text, source_name="正文-第N章", wait=true)
remember(state_changes, source_name="状态-第N章结束", wait=true)
evolve_entity_summary(family_id="ent_XXX")
dream_run(strategy="narrative")

# ===== Phase 4: 伏笔管理 =====
quick_search(query="所有未回收的伏笔")
dream_run(strategy="cross_domain")
dream_run(strategy="contrastive")
```

## 附录：source_name 命名规范

统一的 source_name 前缀让后续检索更精准：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `设定-*` | 世界观设定 | `设定-世界观`, `设定-角色-林嘿嘿` |
| `大纲-*` | 故事大纲 | `大纲-总纲`, `大纲-第3章` |
| `预设-*` | 章节场景预设 | `预设-第3章` |
| `正文-*` | 正式正文 | `正文-第3章-深渊之下`, `正文-第3章-P2` |
| `状态-*` | 章节结束状态快照 | `状态-第3章结束` |
| `伏笔-*` | 伏笔记录 | `伏笔-001-暗河回声` |
| `修改-*` | 设定变更记录 | `修改-角色-林嘿嘿-突破七品` |
