# 大模型 RAG 系统优化实战：从 60% 到 95% 召回率

## 背景

我们在构建企业内部知识库的 RAG（Retrieval-Augmented Generation）系统时，初始版本的召回率只有 60%。经过三个月的迭代优化，最终将召回率提升到了 95%。本文记录整个优化过程。

## 技术栈

- 向量数据库：Milvus 2.3（部署地址 `milvus.internal:19530`）
- Embedding 模型：BGE-large-zh-v1.5（维度 1024）
- LLM：GLM-4-Plus（通过 `https://open.bigmodel.cn/api/paas/v4` 调用）
- 框架：LangChain 0.1.x + 自研 pipeline

## 优化步骤

### 第一步：Chunk 策略优化

原始方案使用固定 512 token 切分，导致很多表格和代码被截断。我们改用语义切分：
- Markdown 按 heading 切分
- 代码按函数/类切分
- 表格保持完整

这一步将召回率从 60% 提升到 72%。

### 第二步：混合检索

纯向量检索对精确匹配（如错误码 `ERR-4021`、版本号 `v2.3.1`）效果差。我们加入了 BM25 稀疏检索，用 RRF（Reciprocal Rank Fusion）融合排序：

```python
def rrf_merge(vector_results, bm25_results, k=60):
    scores = {}
    for rank, doc in enumerate(vector_results):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank + 1)
    for rank, doc in enumerate(bm25_results):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: -x[1])
```

混合检索将召回率从 72% 提升到 85%。

### 第三步：Query 改写

用户查询往往过于简短。我们用 LLM 先将查询改写为更完整的检索语句。例如：
- 用户输入："部署失败怎么办" → 改写为 "Kubernetes 部署失败排查步骤 kubectl rollout undo 回滚"

这一步将召回率从 85% 提升到 92%。

### 第四步：Cross-Encoder 重排

最后用 Cross-Encoder 对 top-20 候选做精排。我们微调了 bge-reranker-large，使用内部标注的 2000 条 query-passage 对。最终召回率达到 95%。

## 总结

优化路径：60% → 72% → 85% → 92% → 95%
关键投入：语义切分 > 混合检索 > Query 改写 > 重排
