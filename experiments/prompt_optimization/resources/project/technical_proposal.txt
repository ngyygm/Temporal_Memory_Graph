# 技术方案：分布式任务调度平台

## 现状与问题
当前使用 crontab + 简单 Shell 脚本管理定时任务，存在以下问题：
1. 单点故障：调度脚本运行在一台机器 `scheduler-prod-01`，宕机即全部中断
2. 无法监控：任务失败无告警，靠人工巡检发现
3. 依赖混乱：任务间依赖靠硬编码 sleep 等待，经常因前置任务延迟导致数据错误
4. 无法回溯：历史执行记录只保留 7 天，排查问题困难

## 方案设计

### 架构选型
对比了三个方案：
- **XXL-Job**：Java 生态，功能完善但与 Python 技术栈不匹配
- **Airflow**：Python 原生，DAG 定义灵活，但重、运维成本高
- **自研轻量方案**：基于 Redis + FastAPI，满足当前 50 个任务规模

选择 **Airflow**（中期）+ **自研轻量方案**（短期过渡）。

### 核心模块

#### 1. 调度器（Scheduler）
- 基于 APScheduler 实现分布式调度
- 任务定义存 PostgreSQL `airflow_db`，表 `dag_run` / `task_instance`
- 支持 cron / interval / one-time 三种触发模式

#### 2. 执行器（Executor）
- KubernetesExecutor：每个任务启动独立 Pod
- 资源隔离：CPU/内存按任务配置限定
- 任务镜像统一管理：`registry.internal/airflow-tasks:{version}`

#### 3. 依赖管理
- DAG 定义任务依赖关系：`task_c >> [task_a, task_b]`（task_c 依赖 a、b 都完成）
- 触发规则：all_success / all_done / one_success

#### 4. 告警与监控
- 任务失败 → Slack 通知 #data-team-alerts
- 超时检测：超时阈值可配置，默认 2 小时
- 每日汇总报告：成功/失败/延迟任务数

## 部署方案
- Airflow Webserver + Scheduler: 2 replicas, K8s namespace=airflow
- PostgreSQL: `airflow-db.internal:5432`, RDS 实例
- Redis: `redis-cluster-01.internal:6379`（复用现有集群）
- 存储: MinIO `minio.internal:9000` 用于任务产物

## 迁移计划
Week 1: 部署 Airflow 环境，迁移 5 个低优先级任务验证
Week 2-3: 迁移剩余 45 个任务，每个任务编写 DAG 定义 + 测试
Week 4: 灰度并行运行，双写验证数据一致性
