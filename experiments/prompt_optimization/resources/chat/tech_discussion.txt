# 描述: 技术团队讨论微服务架构迁移

张明：各位，我们下周要开始把用户服务从单体拆出来了，先说下方案。

李伟：数据库怎么分？用户表和订单表现在是同一个库。

张明：我的想法是先做读写分离，用户服务只读用户表，写操作暂时还走主库。等稳定了再迁移写。

王芳：那注册和登录怎么办？这两个是高频写操作。

张明：注册走新服务，登录还是走老的。登录后面再加个缓存层，Redis 存 session token。

李伟：Redis 集群我们已经有 `redis-cluster-01.internal:6379`，可以直接用。密码在 vault 路径 `secret/infra/redis-prod`。

王芳：API 网关需要改吗？现在的 Kong 配置 `kong.yml` 里路由规则都是按单体写的。

张明：需要改。我打算用 path prefix 来区分：`/api/v2/users/*` 走新服务，`/api/v1/*` 还是走老系统。灰度期间两套并存。

李伟：那监控呢？我们现在的 Grafana dashboard `grafana.internal/d/user-service` 需要加新的 panel。

张明：对，新的 dashboard 叫 `user-service-v2`，我会加上 P99 延迟和错误率的 alert。阈值先设 P99 > 500ms 触发 warning，> 1s 触发 critical。

王芳：时间节点呢？

张明：下周一（2024-11-18）开始，11月底完成拆分，12月第一周灰度 5% 流量，没问题的话12月中旬全量。

李伟：OK，我负责数据库迁移脚本，用 Flyway 管理，migration 文件放 `db/migrations/V2024.11__user_service_split.sql`。

王芳：我负责 Kong 配置和 Redis 接入。提醒一下，`redis-cluster-01` 的 maxmemory-policy 现在是 `allkeys-lru`，session 数据需要注意 TTL 设置。

张明：好，session token TTL 设 7 天，refresh token 存 DB。今天先这样，各自准备，周五 review 方案。
